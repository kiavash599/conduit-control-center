# Dependency lock schema (ADR-0003 Amendment A1)

Three locks pin the dependency closure with hashes; `requirements.txt` stays bounds-based.

- `requirements-aarch64.lock` — PyPI **wheel** sha256; runtime install on aarch64 via the index,
  `pip install --require-hashes --only-binary=:all: -r requirements-aarch64.lock`.
- `requirements-armv7.lock` — resulting **wheel** sha256; runtime install on armv7l OFFLINE,
  `pip install --no-index --only-binary=:all: --require-hashes --find-links <wheelhouse> -r requirements-armv7.lock`.
- `requirements-armv7-build.lock` — PyPI **sdist** sha256; the **build-input** authorization used to
  fetch sources before building armv7 wheels (`pip download --require-hashes -r requirements-armv7-build.lock`).
  This prevents an unapproved source input from being built (finding #6); the post-build provenance
  record then binds each sdist->wheel. Distinct build-input vs runtime-wheel locks avoid hash-any-match
  ambiguity.

**Lifecycle (no circularity):** `requirements-aarch64.lock` and `requirements-armv7-build.lock` are
build-INDEPENDENT and **committed PRE-TAG**. `requirements-armv7.lock` (built-wheel hashes) is
build-DEPENDENT and is a **POST-TAG** content-addressed input injected into the armv7 artifact alongside
the wheelhouse + `provenance/wheelhouse-armv7.json`, and digest-bound.

The `.example` files here are **NON-ACTIVE schema fixtures** (not read by install/update, used only to
unit-test `release/lock_validate.py`). The real locks are generated by `release/gen_locks.py` during
the controlled build and their sha256 digests are bound in the signed manifest (`dependency_locks`,
computed by the producer from the canonical committed bytes).

**Release-input gate:** until the controlled build populates the real root-level `requirements-*.lock`,
the hash-locked install intentionally fails closed. Do NOT merge with active 0.0.0/placeholder locks.
