# Dependency lock schema (ADR-0003 Amendment A1 + A5 dual-origin)

Locks pin the dependency closure with hashes; `requirements.txt` stays bounds-based. Since the
v0.3.17 dual-origin (A5) model there are **two different KINDS** of lock, and confusing them is a
defect: *complete-solution* locks cover the whole 30-package closure, while the *partition* lock
covers only its own origin partition by design.

### Complete-solution locks — validated by `release/lock_validate.py`

- `requirements-aarch64.lock` — PyPI **wheel** sha256; runtime install on aarch64 via the index,
  `pip install --require-hashes --only-binary=:all: -r requirements-aarch64.lock`.
- `requirements-armv7.lock` — resulting **wheel** sha256; runtime install on armv7l OFFLINE,
  `pip install --no-index --only-binary=:all: --require-hashes --find-links <wheelhouse> -r requirements-armv7.lock`.
- `requirements-armv7-solution.lock` — PyPI **sdist** sha256 for the complete **30-package** armv7
  closure. This is the **durable generator INPUT**: the authoritative solution that
  `release/builder/gen_active_inputs.py` partitions. Generated with `release/gen_locks.py`.

### Partition artifacts — NOT complete solutions

- `requirements-armv7-build.lock` — the **source-build partition**: exactly the approved **six**
  packages (`cffi, httptools, markupsafe, psutil, pyyaml, uvloop`) as PyPI **sdist** sha256, the
  build-input authorization used to fetch sources before building (`pip download --require-hashes`).
  It deliberately omits the 24 reused packages, so it must be validated with
  `ccc_release.validate_build_partition_lock()` and **never** with `lock_validate.validate()`, which
  would reject it for "missing" packages that belong to the reuse partition.
- `release/builder/armv7-reuse-authz.json` — the **reuse partition**: exactly the **24** reused
  official wheels by exact identity (see `armv7-reuse-authz.example.json`).

The two partition artifacts are DISJOINT and their union is exactly the 30-package closure.

### Roles at a glance

| file | role | arity | complete solution? | produced by |
|---|---|---|---|---|
| `requirements-aarch64.lock` | aarch64 runtime install | 30 | yes | `gen_locks.py` |
| `requirements-armv7.lock` | armv7 runtime install (offline) | 30 | yes | build (POST-TAG) |
| `requirements-armv7-solution.lock` | durable generator input | 30 | yes | `gen_locks.py` |
| `requirements-armv7-build.lock` | source-build partition | **6** | **no** | `gen_active_inputs.py` |
| `release/builder/armv7-reuse-authz.json` | reuse partition | **24** | **no** | `gen_active_inputs.py` |

### Derived-input state machine

The two derived partition artifacts are co-produced and committed ATOMICALLY, so exactly two states
are legal (enforced by `ccc_release.release_preflight`):

1. **`pre_generation`** — BOTH absent. The only valid state before the Owner runs the generator;
   `python -m release.preflight` (dev mode) is green, release mode is red.
2. **`generated`** — BOTH present, validated as the exact disjoint 6+24=30 partition; green in both modes.
3. **half-state** — exactly one present: INVALID IN EVERY MODE (fails closed, "not atomic").

**Lifecycle (no circularity):** `requirements-aarch64.lock` and `requirements-armv7-solution.lock` are
build-INDEPENDENT and **committed PRE-TAG**; the two derived partition artifacts are then generated from
the solution and committed atomically, also PRE-TAG. `requirements-armv7.lock` (built-wheel hashes) is
build-DEPENDENT and is a **POST-TAG** content-addressed input injected into the armv7 artifact alongside
the wheelhouse + `provenance/wheelhouse-armv7.json`, and digest-bound.

The `.example` files here are **NON-ACTIVE schema fixtures** (not read by install/update; used only to
unit-test the validators). Each is exercised by the validator matching its ROLE:
`requirements-aarch64.lock.example` and `requirements-armv7.lock.example` by
`release/lock_validate.py` (complete-solution — `requirements-armv7-solution.lock` shares this same
complete-solution schema and has no separate fixture), and `requirements-armv7-build.lock.example` by
`ccc_release.validate_build_partition_lock` (six-entry partition). The real complete-solution locks are
generated by `release/gen_locks.py`; the real partition artifacts by
`release/builder/gen_active_inputs.py`. Their sha256 digests are bound in the signed manifest
(`dependency_locks`, computed by the producer from the canonical committed bytes).

**Release-input gate:** until the controlled build populates the real root-level `requirements-*.lock`,
the hash-locked install intentionally fails closed. Do NOT merge with active 0.0.0/placeholder locks.
