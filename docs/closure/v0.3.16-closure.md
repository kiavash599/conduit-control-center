## 1. Objective

Ship one CCC version safely on both supported Raspberry Pi platforms without
making armv7l's offline native-dependency requirement part of the aarch64
artifact. Preserve one publisher identity and one signature ceremony while
preventing the backend, delivery channel, filename, or payload from choosing the
privileged target platform.

## 2. Scope delivered

### V2 release composition

- `ccc-X.Y.Z-aarch64.tar.gz`: deterministic tagged source/runtime tree; no
  wheelhouse.
- `ccc-X.Y.Z-armv7l.tar.gz`: the same source/runtime tree plus the embedded
  `wheelhouse-armhf/`, `SHA256SUMS`, strict provenance, and generated armv7l
  runtime lock.
- `ccc-X.Y.Z.manifest.json` and `.sig`: one canonical V2 manifest and one SSHSIG
  signature binding both mandatory platform entries and their dependency chain.
- A release missing either platform artifact is incomplete and rejected.

### Dependency supply chain

- `requirements-aarch64.lock` is the committed, hashed aarch64 wheel solution.
- `requirements-armv7-build.lock` is the committed, hashed PyPI sdist
  authorization set used to build the armv7l wheelhouse.
- `requirements-armv7.lock` is build-dependent and is generated after the tag
  from the actual wheelhouse; it is injected and digest-bound, not committed as a
  pre-tag placeholder.
- The aarch64 artifact stays wheelhouse-free. The armv7l artifact installs its
  embedded closure with `--no-index --only-binary=:all:`.

### Platform authorization

- The backend uses the detected platform only to discover/download the matching
  signed entry.
- The privileged helper independently evaluates `uname -m`, selects the signed
  entry for the real host, and verifies that the received artifact digest matches
  that entry before apply.
- V1 manifests, unknown platforms, missing/duplicate entries, cross-platform
  artifacts, and any fallback path are rejected fail-closed.

## 3. Security properties

- Existing SSH Ed25519 publisher identity, SSHSIG namespace, and out-of-band
  device trust anchor remain unchanged.
- The canonical signed manifest binds product/version, source tag and commit,
  exact platform artifact names/digests, requirements and pre-tag lock digests,
  and the armv7l runtime-lock/wheelhouse/provenance chain.
- Extraction enforces exact top-level/member allowlists, path and link safety,
  case-fold collision rejection, bounded size/disk checks, secret exclusions,
  and no NUL bytes in text/script members.
- Provenance authorizes each armv7l wheel against the committed sdist build lock;
  the embedded runtime lock must describe the produced wheel closure exactly.
- Verification precedes every privileged action. Audit/logging remains
  non-authorizing.

## 5. Migration and One-Click Update boundary

- V1 compatibility is intentionally retired because no external v0.3.14 clients
  exist. The Owner's legacy v0.3.14 Raspberry Pi is migrated once through the
  internal manual SSH runbook.
- v0.3.14 → v0.3.16 is therefore not represented as a V2 One-Click validation.
- This is the first V2 release, so a real cross-version V2 → V2 dashboard update
  cannot exist until a later V2 release is available. That future validation is
  maintenance evidence, not a reason to weaken or emulate V1 compatibility in
  v0.3.16.
