## 1. Objective
Ship two operator-facing capabilities: (1) choose a Cloudflare-compatible HTTPS
port at install time, and (2) update Conduit Control Center from the dashboard
with one click ÔÇö both secure, beginner-friendly, and safe to run on low-spec
Raspberry Pi hardware. Conduit Core updates remain out of scope.

## 2. Scope delivered

### Feature 1 ÔÇö Cloudflare-compatible HTTPS port selection
- `install.sh` prompts for an HTTPS port from the Cloudflare-supported set
  (`443`, `8443`, `2053`, `2083`, `2087`, `2096`; default `443`), excludes
  occupied ports, and treats the currently-configured port as available on a
  re-run. No custom ports.
- `config.json` `web.https_port` is the single source of truth; `update.sh`
  preserves it (legacy installs infer `443`) and never resets a non-443 install.
- `deployment/bin/ccc-apply-https-port` renders the nginx site + redirect, runs
  `nginx -t`, reloads, and reconciles **only** the `CCC HTTPS`-commented UFW rule.
- The dashboard shows a read-only "HTTPS port (configured)".

### Feature 2 ÔÇö One-click CCC update
- Dashboard "Software Updates" section (Chapter 7 ┬ž7.22): current/latest/status,
  last-checked, release-notes preview, Check Now, Install Update.
- `backend/api/update.py` (`/api/update/check|install|status`): GitHub Releases
  "latest" (stable only ÔÇö drafts/prereleases excluded); 24h cache; GitHub failure
  never breaks the dashboard; recommended-Core warning.
- `deployment/bin/ccc-update-apply` (root, argv-only, flock): payload tarball on
  stdin only; runs the hardcoded installed `update.sh --ccc-only`; double-fork +
  setsid worker; outcome in `/var/lib/conduit-cc/update-status.json`.
- Frontend reconnects across the service restart; automatic rollback via
  `update.sh` Phase 5. No auto-update; Conduit Core untouched.

## 3. Key decisions
- **CCC-only updates** via a new `update.sh --ccc-only` (skips Phase 2b / Conduit
  Core). Core compatibility is a non-blocking warning, operator decides.
- **Stable GitHub Releases** are the only update source ÔÇö never `origin/main`,
  branches, or arbitrary refs; no user-supplied URL/ref/path.
- **Dashboard editing of the port is rejected** (lockout/external-coupling risk);
  port changes stay a local install/CLI action. A management CLI is reserved,
  not implemented.
- Update architecture **mirrors the restore subsystem** (detached worker, async
  status, PID reconciliation, reconnect).

## 4. Security model
- Auth on all update endpoints; **CSRF** on `/install`.
- One sudoers grant per helper (`ccc-apply-https-port`, `ccc-update-apply`),
  `visudo -cf`-validated, re-provisioned by install.sh and update.sh.
- Helpers are argv-only, hardcoded-action, no shell; the update helper accepts
  only `apply` and re-validates the stdin payload (CCC tree, strict version
  increase ÔÇö no downgrade/reinstall); the update never executes payload scripts.
- Release notes sanitized to plain text (no HTML/scripts). Download host-allowlisted.

## 5. Rollback model
`update.sh` Phase 5 auto-rollback (code/conf/pip/unit) restores the previous CCC
version on failure; `.env`, TLS, `config.json`, and the database are never
modified; Conduit Core is untouched. Power-loss leaves recoverable backups under
`/var/backups/conduit-cc`; a stale `in_progress` is reconciled via PID liveness.
No rollback UI in this release (deferred).
