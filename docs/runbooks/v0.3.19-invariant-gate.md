# v0.3.19 — cumulative invariant gate (authoritative execution contract)

The combined Epic 1 + Epic 2 acceptance unit (and every later v0.3.19 gate) is
validated by ONE committed entry point: `tests/invariant_suite.py`. CI runs the
same entry point (`.github/workflows/ci.yml`, job `invariant-suite`). Nothing
else may claim the authoritative marker.

The combined unit includes the privileged ownership boundary, immutable runtime
store, deterministic candidate identity, fresh-install initial activation,
per-attempt write-ahead update/rollback transaction, executable models for all
explicitly supported legacy baselines, exact RPi2/RPi4 device rehearsals, and
the rollback-reserve acceptance lifecycle. No one
subsystem may be qualified separately as a substitute for this integrated gate.

## Marker semantics (structural, not conventional)

- `INVARIANT_SUITE=PASS` — only a platform-complete, unweakened,
  conftest-enabled execution on the MATCHING host can emit this.
- `INVARIANT_SUITE=SMOKE` — any weakening (`--noconftest`, `--smoke-subset`,
  cross-platform execution, dependency-limited environments) is capped at
  SMOKE. SMOKE never satisfies a release gate.
- `INVARIANT_SUITE=FAIL` — inventory, collection, test, Ruff or shell-gate
  failure.

The marker line records: platform, host, interpreter, conftest mode,
inventory/discovered counts, passed/failed/skipped counts, and exit status.
Record the full line in the qualification evidence ledger.

## Authoritative commands

Windows (real Ruff + the documented exact Windows-compatible module list):

    .venv\Scripts\python.exe -m tests.invariant_suite --platform windows

The runner creates and removes a unique attempt-owned pytest `--basetemp`; do
not replace it with a shared `pytest-of-<user>` directory. The final marker
records `errors` separately from `failed`, so setup/collection failures cannot
be reported as a zero-failure test result.

WSL/Linux (COMPLETE unweakened discovery of `tests/unit/` and
`tests/integration/` with real conftest, plus `bash -n` and the exact reviewed
ShellCheck baseline over `release/builder/*.sh`, and zero-warning ShellCheck
over install/update/uninstall, the first-transition bootstrap and operational
scripts):

    python3 -m tests.invariant_suite --platform linux

## Required-invariant inventory (deletion guard)

`tests/invariant_inventory.txt` lists EVERY authoritative `test_*.py` under
`tests/unit/` and `tests/integration/`, sorted, deduplicated. The gate fails
if a listed file is missing, a discovered file is unlisted, or the inventory is
malformed. Adding a test REQUIRES a same-commit inventory entry. Removing an
entry requires an explicit, separately reviewed lifecycle record (commit
message + this runbook's changelog note). Discovery controls execution; the
inventory controls deletion/coverage.

## Platform authority

Windows curation exists ONLY because of platform incompatibility (the four
legacy POSIX modules with bash-only gates and linux-marked suites); every
excluded module runs unweakened in the Linux gate. There is deliberately NO
curated Linux list.

## Integrated acceptance matrix (mandatory before tag/SRT)

The authoritative Windows and Linux markers are necessary but not sufficient.
Record all of the following against the same clean candidate commit:

1. The committed executable legacy-layout model must pass independently for
   `0.3.14`, `0.3.15`, and `0.3.18`. Exact-device rehearsals must start from
   the observed RPi2 `0.3.14` baseline and RPi4 `0.3.15` baseline: candidate
   built before downtime; legacy runtime conversion; trust transition;
   activation; deploy; injected post-activation health failure; selector/code/
   helper restoration; exact old-version health; preserved immutable candidate
   and rollback reserve. The models are not substitutes for the two device
   gates. The unreleased `0.3.18` path remains executable-model qualified; it
   must not be represented as field-device evidence.
2. Every normal transaction phase is interruption-classified: pre-downtime
   phases become `diagnostic_failure`; post-downtime phases finish the ordered
   resumable rollback and become `rolled_back`; no incomplete attempt remains.
3. Fresh install on an empty target builds and validates a deterministic runtime
   before `activate-initial`; dependency failure leaves the selector absent.
4. Normal signed update succeeds; pre-downtime dependency failure leaves the
   service and active selector untouched; activation and post-activation health
   failures restore the exact previous healthy version without pip/network.
5. RPi2 uses only the signed 30-wheel offline closure. RPi4 uses the signed
   hash-locked aarch64 dependency lock. Neither platform upgrades pip at install
   or update time.
6. Trust, `.env`, ownership, sudoers/helper exact-set, symlink/substitution,
   source-identity and secret-hygiene negative matrices all fail closed.
7. Restore and update mutual exclusion is exercised in both directions. The
   restore API receives its exact ack before service shutdown, secrets traverse
   only the root-owned FIFO handoff, stopping `conduit-cc.service` kills its full
   control group, and interruption at every pre-ack/post-ack boundary leaves no
   unrecorded work tree or overlapping lifecycle operation.

The committed executable rehearsal over production runtime/transaction
primitives and the static shell wiring contracts run inside the invariant
suite. Real service/systemd/network behavior remains device-authoritative and
must be run on both RPi2 and RPi4.

## Supported legacy baseline binding

The first-transition ceremony accepts exactly `0.3.14`, `0.3.15`, or `0.3.18`.
The Owner MUST pass the independently diagnosed device baseline explicitly:

```bash
--expected-installed-version 0.3.14   # RPi2 observed baseline
--expected-installed-version 0.3.15   # RPi4 observed baseline
```

Use only the line matching the target device. The bootstrap rejects every
other value and compares the installed `_version.py` before creating any
update state. The staged engine independently parses the installed version,
requires the same argument for a bootstrap runtime, compares it before the
transaction begins, and repeats the comparison after interrupted-transaction
reconciliation. The schema-2 rollback-reserve record persists the expected
baseline; `reserve-ready` additionally requires the successful transaction's
immutable `previous_version` fact to match it. A mismatch at any layer fails
closed. Neither the installed legacy updater nor installed Python application
code is executed to make this decision.

## First-transition reserve acceptance

`ccc-bootstrap.sh` prints the exact `BOOTSTRAP_ATTEMPT_ID`, source commit and
source tag. Preserve them together with the expected/observed installed
version in evidence. The staging rollback reserve MUST remain present through
all device checks. Only after the combined acceptance matrix is green may the
Owner run:

```bash
sudo /usr/bin/python3 -I /opt/conduit-cc/bin/ccc-runtime \
  reserve-accept <BOOTSTRAP_ATTEMPT_ID> <40-hex-source-commit> v0.3.19
```

Success emits a `BOOTSTRAP_RESERVE` JSON record whose state is `accepted` and
whose history ends `ready, acceptance_intent, accepted`. The operation validates
the exact signed source identity, writes intent before deletion, deletes only
the recorded direct-child staging directory, and is idempotent after a crash.
Do not use `rm -rf`, a wildcard, a prefix sweep, or any manual substitute.

### Failed pre-downtime bootstrap reserve

A failed bootstrap reserve is not accepted and MUST NOT be removed manually.
First preserve the transcript, reserve record, transaction record, health
evidence and source identity outside the device. A corrected bootstrap may use
a new attempt id while the failed reserve remains present. After the corrected
v0.3.19 runtime is installed, the Owner may remove the old reserve only with:

```bash
sudo /usr/bin/python3 -I /opt/conduit-cc/bin/ccc-runtime \
  reserve-discard-failed <FAILED_ATTEMPT_ID> <FAILED_40_HEX_SOURCE_COMMIT> v0.3.19
```

The command succeeds only when the immutable transaction is
`diagnostic_failure`, its history never reached `downtime_started`, source/tag/
target/baseline bindings match the reserve, and the work path is the exact
root-owned mode-0700 recorded directory. It writes
`failure_discard_intent` before deletion and ends in `failure_discarded`, so a
crash is resumable. It cannot discard a successful, ready, accepted,
rolled-back, post-downtime, substituted or foreign reserve. Record the emitted
`BOOTSTRAP_RESERVE` JSON and verify that the old transaction record remains as
durable failure evidence.
